# Visual Album Studio — Developer Readme

This repo builds a local-first desktop editor that generates looping, music-synced videos (10s → 2h+) with a deterministic, resumable export pipeline and optional YouTube publishing later.

The documentation in `docs/**` is the specification.

## Quickstart

### Prerequisites

- **Godot Engine 4.4.x** (used for UI + preview + deterministic offline rendering via MovieWriter).
- **FFmpeg** (used as the export/encode/mux "truth pipeline"; managed/pinned by repo scripts).
- **Python 3.11+** (audio analysis worker using librosa; dependencies pinned).
- **Rust stable toolchain** (small keychain helper via `keyring` crate; Phase 5+).

> Note: Exact pinned versions and checksums are defined by the repo in `tools/versions.*` and enforced by `./scripts/bootstrap.*`.

### Setup (first time)

```bash
# 1) Bootstrap toolchain (downloads/validates pinned FFmpeg, sets up venv, etc.)
./scripts/bootstrap.sh

# 2) Run unit + integration tests
./scripts/test/unit.sh
./scripts/test/integration.sh

# 3) Launch the editor
./scripts/dev/run_editor.sh
```

## First successful Phase 1 bundle (walkthrough)

This is the canonical "we shipped Phase 1" path.

1. Launch the editor.
2. Create a new project:
   - Visual mode: **Motion Poster (G)**
   - Export preset: **1080p30**
   - Duration: **10 seconds** (start small)
3. Import assets:
   - Audio: import an MP3/WAV/FLAC/etc.
   - Album art image: import a PNG/JPG.
4. Verify audio analysis:
   - BPM and beat grid appear in the project panel.
   - Analysis status shows "cached" after completion.
5. Choose a Motion Poster preset:
   - Pick 1 of the 6+ presets.
   - Confirm preview looks correct.
6. Set license provenance for audio:
   - Open the audio asset record.
   - Fill required fields (source type, license notes, attribution/proof if needed).
7. Export production bundle:
   - Click **Export → Production Bundle**.
   - Observe queue progress (render segments → encode → concat → bundle).
8. Verify outputs:
   - Navigate to the bundle directory (see Output Conventions below).
   - Confirm the bundle contains:
     - `video.mp4`
     - `thumbnail.png`
     - `metadata.json`
     - `provenance.json`
     - `build_manifest.json`

## Running phase acceptance checks locally

Phase gates are mandatory. Do not proceed to later phases until the current phase passes.

```bash
# Phase 1 gate
./scripts/test/acceptance_phase_01.sh

# Later phases (after implemented)
./scripts/test/acceptance_phase_02.sh
./scripts/test/acceptance_phase_03.sh
./scripts/test/acceptance_phase_04.sh
./scripts/test/acceptance_phase_05.sh
./scripts/test/acceptance_phase_06.sh
./scripts/test/acceptance_phase_07.sh
```

Expected behavior:
- Acceptance scripts create outputs under `out/` (gitignored).
- Scripts must exit **0** on success and **non-zero** on failure.
- Failures must print actionable diagnostics and point to logs/artifacts.

## Output directory conventions

Nothing generated by running the app or tests should be committed to git.

Repository-local (developer/test) outputs:
- `out/`  
  - `out/tmp/` — job workspaces (segments, intermediate frames, partial encodes)
  - `out/exports/` — finalized production bundles from test runs
  - `out/logs/` — logs (redacted; safe for bug reports)

End-user app data (installed app):
- macOS: `~/Library/Application Support/VisualAlbumStudio/`
- Windows: `%APPDATA%\VisualAlbumStudio\`
- Linux: `~/.local/share/VisualAlbumStudio/`

Inside the app data directory:
- `vas.db` — SQLite DB
- `library/` — managed asset library (hash-addressed)
- `models/` — optional ML model weights (Phase 2; user opt-in)
- `exports/` — user exports
- `tmp/` — workspaces
- `logs/` — logs
